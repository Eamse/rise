# RISE AUTOPARTS B2C 도메인 설계 (v3)

## 1) 전체 구조 (왜 필요한가)
`B2C 쇼핑몰`은 핵심 흐름이 명확해야 운영이 쉽다.

```text
[회원 유입]
    ├── 회원가입/로그인 (Identity)
    └── 비회원 탐색

[상품 탐색]
    ├── 카테고리/검색 (Catalog)
    ├── 상품 상세
    └── 찜/리뷰 확인 (Engagement)

[구매 진행]
    ├── 장바구니 담기 (Cart)
    ├── 주문 생성 (Order)
    ├── 결제 승인 (Payment)
    └── 주문 상태 갱신

[배송/완료]
    ├── 출고 처리 (Fulfillment)
    ├── 배송 추적
    └── 구매확정 + 리뷰 작성
```

## 2) 현재 기준 (As-Is)
```text
[현재 구현됨]
    ├── User / Admin / Address / PaymentMethod
    ├── Product
    ├── CartItem
    └── Order / OrderItem

[현재 부족함]
    ├── Payment / Refund (결제 이력)
    ├── Shipment / ShipmentEvent (배송 이력)
    ├── Coupon / UserCoupon (프로모션)
    ├── WishlistItem / Review (전환율 기능)
    └── InventoryHistory / AdminActionLog (운영 추적)
```

## 3) 도메인 맵 (Where + 분리 이유)
```text
[Identity]
    ├── User
    ├── Address
    └── 인증/권한

[Catalog]
    └── Product

[Cart]
    └── CartItem

[Order]
    ├── Order
    └── OrderItem

[Payment]
    ├── Payment
    └── Refund

[Fulfillment]
    ├── Shipment
    └── ShipmentEvent

[Promotion]
    ├── Coupon
    └── UserCoupon

[Engagement]
    ├── WishlistItem
    └── Review

[AdminOps]
    ├── InventoryHistory
    └── AdminActionLog
```

## 4) 구매 플로우 상세 (데이터가 어떻게 흐르는가)
```text
[상품 선택]
    ├── Product 조회
    └── CartItem 생성/수정

[주문 생성]
    ├── CartItem -> OrderItem 스냅샷 복사
    ├── Order.totalPrice 계산
    └── Order.status = PENDING

[결제 승인]
    ├── Payment 생성 (provider/paymentKey/status)
    ├── 성공: Order.status = PAID
    └── 실패: Payment.status = FAILED

[배송 처리]
    ├── Shipment 생성 (carrier/trackingNumber)
    ├── ShipmentEvent 누적
    └── 완료: Order.status = DELIVERED
```

## 5) 주문 상태 (OrderStatus)
```text
PENDING
    ├── 결제 성공 -> PAID
    └── 취소 -> CANCELED

PAID
    ├── 상품 준비 -> PREPARING
    └── 환불 -> REFUNDED

PREPARING
    └── 출고 -> SHIPPED

SHIPPED
    └── 배송 완료 -> DELIVERED
```

## 6) API 설계 원칙 (중복 방지)
```text
[원칙]
    ├── 같은 리소스는 하나의 URI 체계만 사용
    ├── 같은 상태 변경을 두 API에서 동시에 수행 금지
    ├── 조회(GET) / 변경(POST,PATCH,DELETE) 분리
    └── UI 편의용 API 이름(add/update 등) 남발 금지
```

### 6-1) 금지 예시
```text
금지
    ├── POST /api/order
    └── POST /api/orders   (의미 중복)

금지
    ├── PATCH /api/orders/:id/status
    └── POST /api/orders/:id/ship   (상태 변경 책임 중복)
```

### 6-2) 권장 API 맵 (v1)
```text
/api/v1/products
    ├── GET /
    └── GET /:id

/api/v1/admin/products
    ├── POST /
    ├── PATCH /:id
    └── DELETE /:id

/api/v1/cart/items
    ├── GET /
    ├── POST /
    ├── PATCH /:id
    └── DELETE /:id

/api/v1/orders
    ├── POST /
    ├── GET /
    ├── GET /:id
    └── PATCH /:id/cancel

/api/v1/payments
    ├── POST /confirm
    └── POST /refunds

/api/v1/shipments
    ├── GET /:orderId
    └── PATCH /:id

/api/v1/coupons
    └── POST /apply

/api/v1/reviews
    └── POST /
```

## 7) Next.js 재활용 구조 (Reusability)
```text
src/
├── app/
│   └── api/
│       └── v1/
│           └── .../route.ts      <-- HTTP 입출력만
└── server/
    ├── services/                 <-- 비즈니스 로직
    ├── repositories/             <-- Prisma 접근
    ├── schemas/                  <-- 입력 검증
    ├── auth/                     <-- 인증/인가
    └── lib/                      <-- 공통 유틸
```

### 7-1) 실행 순서
```text
route.ts
    ├── schema 검증
    ├── service 호출
    ├── repository DB 처리
    └── 표준 응답 반환
```

## 8) 코드 가독성 규칙 (Readability)
```text
[네이밍]
    ├── userId, productId, orderId 일관 유지
    └── 상태값은 enum 사용

[응답 포맷]
    └── { success, data, error } 고정

[함수 크기]
    ├── route handler 60줄 내외
    └── 길어지면 service로 분리

[예외 처리]
    ├── service: throw
    └── route: try/catch + 에러 매핑
```

## 9) 실제 서비스 운영 기준 (Production)
```text
[배포]
    ├── API 버전: /api/v1
    └── Breaking change: /api/v2

[DB]
    ├── migration + rollback 전략
    └── 금액은 정수(원) 저장

[보안]
    ├── 관리자 API 서버 권한검사 필수
    └── rate limit 적용

[관측성]
    ├── 주문/결제/환불/배송 traceId 기록
    └── AdminActionLog 저장

[테스트]
    ├── 주문/결제/환불 통합 테스트
    └── 상태 전이 테스트 (OrderStatus)
```

## 10) 단계별 적용 (Roadmap)
```text
Phase 1 (필수)

    ├── Coupon / UserCoupon
    └── WishlistItem / Review

Phase 3 (운영안정)
    ├── InventoryHistory
    └── AdminActionLog
```
